📶2. Offline-First Mobile App Architecture

What is Offline-First?

Definition: Offline-first apps function seamlessly without an active internet connection by storing data locally first and syncing with servers when online. Local data storage is the source of truth.

Why Offline-First for Prayer Tracking?

CRITICAL for Prayer Apps:
  • Prayer times must work offline - Kids need access even in low connectivity areas
  • Track prayers anytime - Mark prayers as complete without internet
  • Improved UX - No loading spinners, instant interactions
  • Airplane mode support - Works during travel

Core Components

  1. Local Database - Store and query data instantly (WatermelonDB, MMKV, SQLite)
  2. Sync Engine - Queue user actions/updates, sync with server in background
  3. Network Detection - Use @react-native-community/netinfo to detect connectivity changes
  4. Conflict Resolution - Handle optimistic or pessimistic locking for offline modifications

Architecture Pattern

// Offline-First Data Flow 1. USER ACTION (e.g., mark prayer as complete) ↓ 2. UPDATE LOCAL DATABASE (instant response to user) ↓ 3. ADD TO SYNC QUEUE (store pending server update) ↓ 4. NETWORK DETECTION ├─→ ONLINE: Sync queue to server → Clear queue └─→ OFFLINE: Keep in queue until online // Libraries to Consider: - React Native Offline - Redux Offline (if using Redux) - React Query + AsyncStorage (for caching)

Implementation with React Query

// Example: Offline-First Prayer Tracking with React Query import { useQuery, useMutation } from '@tanstack/react-query'; import AsyncStorage from '@react-native-async-storage/async-storage'; import NetInfo from '@react-native-community/netinfo'; // Hook to track prayer completion const usePrayerTracker = () => { // Query prayers from local storage first const { data: prayers } = useQuery({ queryKey: ['prayers'], queryFn: async () => { const local = await AsyncStorage.getItem('prayers'); return local ? JSON.parse(local) : []; }, staleTime: Infinity, // Local data never goes stale }); // Mutation to mark prayer as complete const markComplete = useMutation({ mutationFn: async (prayerId) => { // Update local storage immediately const updated = prayers.map(p => p.id === prayerId ? { ...p, completed: true } : p ); await AsyncStorage.setItem('prayers', JSON.stringify(updated)); // Check if online const netInfo = await NetInfo.fetch(); if (netInfo.isConnected) { // Sync to server await api.updatePrayer(prayerId, { completed: true }); } else { // Add to sync queue await addToSyncQueue({ type: 'prayer_complete', prayerId }); } }, }); return { prayers, markComplete }; };
RECOMMENDED APPROACH: For a prayer tracking app, use React Query + AsyncStorage for simple offline-first functionality. If you need complex queries or large datasets (100+ records), upgrade to WatermelonDB.

💾3. Data Persistence Strategies

Decision Guide

Use Case Recommended Solution Why
Simple key-value (settings, theme) MMKV 30x faster than AsyncStorage, simple API
Sensitive data (auth tokens) expo-secure-storage Encrypted storage, dedicated secure place
Small datasets (<100 records) AsyncStorage / MMKV Simple, no overhead, easy to use
Large datasets (100+ records) WatermelonDB Fast queries, lazy loading, reactive
Complex relational data SQLite or WatermelonDB SQL queries, relationships, indexing
NoSQL large datasets Realm 10x faster than SQLite, object-oriented

AsyncStorage Best Practices

WHEN TO USE:
  • Small data that needs serialization
  • Language or theme settings
  • Offline mode support (cached API responses)
  • Limit: 6 MB on Android, unlimited on iOS (design for 6 MB cross-platform)
// AsyncStorage Best Practices // ✅ DO: Create a repository pattern class SettingsRepository { async getTheme() { const theme = await AsyncStorage.getItem('theme'); return theme ? JSON.parse(theme) : 'light'; } async setTheme(theme) { await AsyncStorage.setItem('theme', JSON.stringify(theme)); } } // ✅ DO: Always serialize/deserialize const settings = { language: 'ar', notifications: true }; await AsyncStorage.setItem('settings', JSON.stringify(settings)); const retrieved = JSON.parse(await AsyncStorage.getItem('settings')); // ❌ DON'T: Store sensitive data (tokens, passwords) // Use expo-secure-storage instead // ❌ DON'T: Store large datasets // One app slowed down with 100+ records - use SQLite/WatermelonDB

WatermelonDB for Offline-First

RECOMMENDED for Prayer Tracking: WatermelonDB is ideal for apps that need to work offline and sync later. It's optimized for mobile devices and 10x faster than traditional approaches.

🍉WatermelonDB - Reactive Offline-First Database

✅ Pros

  • Lazy loading - only loads data when needed
  • Fast queries on SQLite (separate native thread)
  • Reactive - auto-rerenders on data changes
  • Offline-first with built-in sync
  • Multiplatform (iOS, Android, web, Node.js)
  • Scales from 100s to 10,000s of records

❌ Cons

  • Not available in Expo managed workflow (requires EAS build)
  • Requires setup plugin for Expo
  • Steeper learning curve than AsyncStorage
  • Overkill for very simple apps

WatermelonDB Schema Example (Prayer Tracker)

// model/schema.js import { appSchema, tableSchema } from '@nozbe/watermelondb'; export default appSchema({ version: 1, tables: [ tableSchema({ name: 'prayers', columns: [ { name: 'name', type: 'string' }, // Fajr, Dhuhr, etc. { name: 'time', type: 'number' }, // Unix timestamp { name: 'completed', type: 'boolean' }, { name: 'completed_at', type: 'number', isOptional: true }, { name: 'points_earned', type: 'number' }, { name: 'date', type: 'string', isIndexed: true }, // For queries ], }), tableSchema({ name: 'rewards', columns: [ { name: 'title', type: 'string' }, { name: 'description', type: 'string' }, { name: 'points_required', type: 'number' }, { name: 'unlocked', type: 'boolean' }, ], }), ], }); // model/Prayer.js import { Model } from '@nozbe/watermelondb'; import { field, date } from '@nozbe/watermelondb/decorators'; export default class Prayer extends Model { static table = 'prayers'; @field('name') name; @date('time') time; @field('completed') completed; @date('completed_at') completedAt; @field('points_earned') pointsEarned; @field('date') date; }

SQLite for Encryption

OPTIONAL: If you need encrypted local storage, use SQLite with SQLCipher extension. SQLite doesn't encrypt by default, but SQLCipher adds AES-128/256 encryption.

🕌4. Prayer Times API Integration

Top Prayer Times APIs (2025)

⭐1. Aladhan API - RECOMMENDED

URL: https://aladhan.com/prayer-times-api

✅ Pros

  • Free and open-source
  • No API key required
  • No authentication
  • Returns JSON with five prayer times
  • Supports location (city, country, address)
  • Monthly prayer times endpoint
  • Multiple calculation methods

⚠️ Considerations

  • Rate limits (check docs for current limits)
  • Requires internet for initial fetch
  • Cache responses for offline use

2. IslamicAPI.com

URL: https://islamicapi.com/doc/prayer-time/

  • Prayer times, Qibla direction, prohibited times
  • Multiple calculation methods (Jafari, Karachi, ISNA, MWL, Umm Al-Qura, Egyptian)
  • Supports regional methods

3. MuslimSalat.com API

URL: http://www.muslimsalat.com/api/

  • Requires API key (registration needed)
  • Location-based prayer times

Calculation Method Support

CRITICAL: Accommodate differences in prayer timing across Islamic schools of thought
  • Standard Method - Shafi, Maliki, Hanbali
  • Hanafi Method
  • Regional Methods:
    • Umm Al-Qura (Makkah) - Recommended for Saudi Arabia
    • Islamic Society of North America (ISNA)
    • University of Islamic Sciences, Karachi
    • Egyptian General Authority of Survey
    • Muslim World League (MWL)

Best Practices for Islamic App Development

1. Location and Time Handling
  • Support Daylight Saving Time adjustments
  • Auto-sync with user's device location or allow manual input
  • Sync prayer times with device's time zone
  • Option for city-specific time zone (account for regional differences)
2. Offline Functionality
  • Fetch prayer times for entire month and cache locally
  • Real-time location updates when online
  • Fallback to cached data when offline
  • Reliable even in remote or low-connectivity areas
3. Customization
  • Automatic default settings or manual customization
  • High level of customization for each of five daily prayer notifications
  • Ability to adjust prayer time offsets (±15 minutes)
  • Manual prayer logging or automatic based on time

Implementation Example (Aladhan API)

// services/prayerTimes.js import AsyncStorage from '@react-native-async-storage/async-storage'; const ALADHAN_API = 'https://api.aladhan.com/v1'; class PrayerTimesService { // Fetch prayer times for entire month async fetchMonthlyPrayerTimes(latitude, longitude, method = 4) { try { const date = new Date(); const month = date.getMonth() + 1; const year = date.getFullYear(); const response = await fetch( `${ALADHAN_API}/calendar/${year}/${month}?latitude=${latitude}&longitude=${longitude}&method=${method}` ); const data = await response.json(); // Cache for offline use await AsyncStorage.setItem( 'monthly_prayer_times', JSON.stringify(data.data) ); return data.data; } catch (error) { // Fallback to cached data const cached = await AsyncStorage.getItem('monthly_prayer_times'); return cached ? JSON.parse(cached) : null; } } // Get today's prayer times async getTodayPrayerTimes() { const monthlyData = await this.fetchMonthlyPrayerTimes( 24.7136, 46.6753 // Riyadh coordinates - get from device location ); const today = new Date().getDate(); const todayData = monthlyData.find(day => parseInt(day.date.gregorian.day) === today ); return { fajr: todayData.timings.Fajr, dhuhr: todayData.timings.Dhuhr, asr: todayData.timings.Asr, maghrib: todayData.timings.Maghrib, isha: todayData.timings.Isha, }; } } export default new PrayerTimesService();

Alternative: Adhan.js Library

OPTION: Use the adhan npm library for local prayer time calculation (no API required)
  • Install: npm install adhan

  • No internet required after installation
  • Smaller app size (no API dependency)

🎮5. Gamification for Children (Ages 7-10)

Key Principles

CRITICAL SUCCESS FACTORS:
  • Fun-to-Learning Ratio - Too much gamification = focus on points, not learning. Too little = lose interest.
  • Align with Learning Objectives - Game mechanics must support lesson goals (prayer consistency, understanding importance)
  • Age-Appropriate Design - Simple rewards, bright visuals, cartoon characters, sound effects
  • Intrinsic Reward Loops - Unlock more content by playing and learning

Effective Gamification Elements

Element Description Implementation for Prayer Tracking
Points Numeric rewards for completing actions Earn 10 points per prayer completed, 50 bonus for completing all 5 daily prayers
Badges Visual achievements for milestones "7-Day Streak", "Early Bird (Fajr)", "Consistency Champion" badges
Levels Progression tiers with increasing difficulty Level 1: Beginner (0-100 pts), Level 2: Committed (100-500 pts), etc.
Streaks Consecutive days of achievement Prayer streak counter - resets if miss a day
Progress Bars Visual representation of completion Daily progress (5 prayers), weekly progress, monthly progress
Unlockables New content unlocked by progress Avatar customizations, new prayer mat designs, Islamic stories
Leaderboards Friendly competition (use carefully) Family leaderboard only - avoid public comparisons for kids

Research-Backed Benefits

Why Gamification Works for Ages 7-10:
  • Accelerated Learning - Kids achieve in weeks what might take months with traditional methods
  • Brain Synapse Creation - Dr. Karyn Purvis research: 400 repetitions → 20 repetitions when learning is done in play
  • Reduced Stress - Gamification reduces learning-associated stress
  • Cognitive Development - Develops cognitive functions through interactive engagement

Best Subject Areas for Gamification

OPTIMAL FOR: Skills that need lots of practice
  • Prayer times memorization (Fajr, Dhuhr, Asr, Maghrib, Isha)
  • Prayer movements (Rakat practice)
  • Arabic vocabulary (prayer recitations)
  • Consistency habits (daily prayer tracking)

Age-Appropriate Visual Design

Visual Elements for Ages 7-10:
  • Bright, vibrant colors - Primary colors, high contrast
  • Cartoon characters - Friendly mascot to guide journey
  • Animated feedback - Celebrations when prayer marked complete
  • Sound effects - Positive audio feedback (keep volume adjustable)
  • Simple icons - Large, clear, recognizable symbols
  • Progress visualization - Stars filling up, trees growing, etc.

Balance and Safety

CRITICAL CONSIDERATIONS:
  • Screen Time Balance - Encourage offline prayer activities, not just app usage
  • No Public Leaderboards - Family-only comparisons to avoid social pressure
  • Inclusive Design - Customization for different learning styles and abilities
  • Parental Controls - Parents can view progress, adjust difficulty
  • No In-App Purchases - All unlocks earned through prayer completion

♿6. Accessibility for Young Users

WCAG Standards for Mobile Apps

LEGAL REQUIREMENT: WCAG 2.1 Level AA is the compliance standard for ADA (Americans with Disabilities Act)
  • Originally for web, now applies to mobile apps (hybrid and native)
  • Mobile-specific guidelines introduced in WCAG 2.1 (June 2018)
  • 17 new success criteria for mobile accessibility

The Four POUR Principles

Principle Description Implementation for Kids App
Perceivable Information must be presentable to users in ways they can perceive • Sufficient color contrast (4.5:1 for normal text)
• Text resizing and reflow support
• Visual + audio feedback for actions
Operable UI components must be operable by all users • Large touch targets (min 44x44 points)
• Clear instructions for gestures
• Consistent navigation
Understandable Information and UI must be understandable • Simple language (age 7-10 reading level)
• Clear button labels
• Immediate feedback on actions
Robust Content must work with assistive technologies • Screen reader support
• Semantic labels for all elements
• Works with TalkBack/VoiceOver

Key Technical Requirements

1. Color Contrast
  • Normal text (<18pt or 14pt bold): 4.5:1 contrast ratio
  • Large text (≥18pt or 14pt bold): 3:1 contrast ratio
  • Test with tools like WebAIM Contrast Checker
2. Touch Targets
  • Minimum size: 44x44 points (iOS) / 48x48 dp (Android)
  • Larger is better for young children (consider 60x60 points)
  • Adequate spacing between interactive elements
3. Screen Reader Support
  • Add accessibilityLabel to all interactive elements
  • Use accessibilityHint to explain what happens
  • Group related content with accessibilityRole
// React Native Accessibility Example import { TouchableOpacity, Text, View } from 'react-native'; const PrayerButton = ({ prayerName, onPress, completed }) => ( {prayerName} );
4. Orientation Support
  • Don't restrict to portrait or landscape unless essential
  • Content should adapt to both orientations
5. Text Descriptions
  • Provide text descriptions for images, icons, buttons
  • Screen readers should describe all non-text elements

Accessibility for Children Specifically

ADDITIONAL CONSIDERATIONS:
  • Larger UI elements - Kids have less precise touch than adults
  • Simple language - Avoid complex words, use age-appropriate vocabulary
  • Visual + audio feedback - Multiple sensory confirmations
  • Forgiving interactions - Easy to undo mistakes
  • Reduced cognitive load - One primary action per screen

Current State of Mobile Accessibility

INDUSTRY STATS (2025):
  • 72% of mobile apps tested had accessibility barriers (State of Mobile App Accessibility report)
  • Common issues: poor screen reader support, low contrast, small touch targets, motion-triggered problems
  • Your app can stand out by being accessibility-first from day one

Testing Checklist

✅ ACCESSIBILITY TESTING CHECKLIST [ ] Color contrast meets 4.5:1 minimum (use WebAIM checker) [ ] All interactive elements are 44x44 points minimum [ ] Screen reader (VoiceOver/TalkBack) can navigate entire app [ ] All buttons have accessibilityLabel and accessibilityHint [ ] Images have meaningful accessibilityLabel [ ] App works in both portrait and landscape [ ] Text can be resized up to 200% [ ] No information conveyed by color alone [ ] Haptic/audio feedback for important actions [ ] Forms have clear labels and error messages [ ] Kids can complete tasks without adult help
